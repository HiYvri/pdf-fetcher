# JLSS Network Analysis

Last updated: 2026-07-01

## Confirmed Facts

- Logged-in Chrome session can access both `helpCenter` and `userCenter`.
- Frontend API base URL is `https://api.jlss.vip`.
- Frontend authentication uses local `token` and sends it as request header `token`.
- Login-invalid signal is response `code === 501`; the frontend clears local state and redirects to `/login`.
- Metadata input for the Zotero plugin is fixed as `DOI > title`.

## Live Validation Evidence

- Test query: a DOI-style article identifier
- Live submit date: confirmed during local manual testing
- Browser result after submit: success dialog `文献传递成功`
- New `userCenter` row observed:
  - content: matched the submitted query
  - task number: generated by the JLSS service
  - time: generated by the JLSS service
  - status: `成功`
  - initial action: `下载资源`
- Live download behavior:
  - clicking `下载资源` opened a signed direct PDF URL on JLSS object storage
  - action label changed to `再次下载`

## Active Interface Mapping

### Submit

- Frontend route: `helpCenter`
- Endpoint: `POST /search/trans`
- Request body:

```json
{
  "content": "<query text>"
}
```

- Success condition used by frontend: response `code === 200`
- Known uncertainty: exact success response payload fields are not needed for initial submit, because the UI later obtains task details from the list endpoint.

### User Center List

- Frontend route: `userCenter`
- Endpoint: `POST /task/myHelpList`
- Request body:

```json
{
  "currentPage": 1,
  "pageSize": 10,
  "data": {
    "taskStatus": "",
    "startTime": "",
    "endTime": ""
  }
}
```

- Fields rendered by the frontend:
  - `taskTitle`: delivery content
  - `taskCode`: task number
  - `createTime`: delivery time
  - `taskStatus`: numeric status
  - `uuid`: task UUID used for download
  - `resourceUuid`: resource UUID used for download
  - `taskMarks`
  - `taskRetain`
- Observed status mapping in frontend:
  - `2`: success
  - `3`: failed
  - `6`: reported/error
  - other values: shown as manual lookup in progress

### Download

- Frontend action: `getResourceUrl(resourceUuid, uuid)`
- Endpoint: `POST /task/clickDownload`
- Request body:

```json
{
  "resourceUuid": "<resourceUuid from list row>",
  "uuid": "<uuid from list row>"
}
```

- Frontend success condition:
  - outer response `code === 200`
  - nested response `data.code === 0`
  - signed direct PDF URL at `data.data`
- The signed URL points to `jlssos.jlss.vip` and is opened directly by the browser.

## Legacy Or Non-Active Candidates

The current frontend bundle also contains these `thesisOrder` endpoints, but they are not the active `helpCenter`/`userCenter` flow verified with the test DOI:

- `POST /web/thesisOrder/add`
- `POST /web/thesisOrder/pageList`
- `POST /web/thesisOrder/cancel`
- `POST /web/thesisOrder/userDelete`

The bundle also contains a `user/hcode` blob-response clue. It is retained as a legacy fallback candidate, but the verified download flow is `/task/clickDownload` returning a signed PDF URL.

## Plugin Implementation Consequences

- Use `token` as the primary auth value.
- Do not make Cookie the primary implementation path unless later evidence proves it is required.
- Submit only one query string per item, selected by `DOI > title`.
- Polling should match remote tasks by `taskTitle` first, then use `uuid` and `resourceUuid` for download.
- Download should call `/task/clickDownload` first, then fetch the returned signed PDF URL without sending the JLSS token to object storage.
